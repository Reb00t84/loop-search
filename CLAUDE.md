## Гипотеза в трёх строках

Квантовый отскок коллапса (bounce) стандартно выбрасывает материю в будущее
белодырной фазой. Здесь рассмотрена обратная склейка: горловина замыкается на
причинное ПРОШЛОЕ той же массы — выброс воссоздаёт облако, из которого дыра
схлопнулась. Замкнутая самосогласованная петля: облако → коллапс → дыра →
отскок → назад → облако. Без экзотической материи (всё запрещённое — внутри
планковской области Q, где полуклассика не определена), без остатка.

## Ключевые следствия (зафиксированы в препринте)

- Петлю замыкает только термализованная КГП (критерий Лоссева–Новикова:
  максимум энтропии = нечему деградировать за оборот). Новиков — не постулат,
  а автоматическое свойство. Барионы на петле — джинн-материя без бариогенезиса.
- Энергобаланс: петля излучает (GRB + GW), масса в горловине сохраняется →
  облако обязано аккрецировать ΔM = εM; ρ_min = 3ε/(4πGT²) ≈ 2ε·ρ_crit(эпохи).
  Изолированные петли невозможны; петли — феномен ранней плотной вселенной.
- Внешняя хронология (диаграмма Пенроуза): t₁ вспышка (EM+GW транзиент без
  прародителя, Биркгоф требует GW-компоненту) → облако (аккреция, T_cloud
  ~10⁶–10⁸ лет) → t₂ коллапс (GW-всплеск без рингдауна, горизонт не
  финализируется) → дыра живёт τ → t₃ беззвучное исчезновение (масса уходит
  в горловину). Интервал GRB→GW = T_cloud (НЕ время жизни дыры!);
  интервал GW→исчезновение = τ напрямую.
- τ(M) — свободная функция (закон отскока никем не выведен). Потолок Хокинга
  τ ≤ M³, динамический пол τ ≥ GM/c³. Дискриминатор против планковских звёзд
  Ровелли (τ ∝ M²): при M² ни один объект тяжелее ~6×10²⁵ г не отскакивает за
  возраст Вселенной → одно зарегистрированное исчезновение массивной ЧД
  фальсифицирует M² целиком.
- Джинн-газ: рехадронизация выброса при S/B ~ 10² (η ~ 5×10⁻² против
  космологических 6×10⁻¹⁰) → дейтерий выгорает, гелий усилен. Наблюдаемое
  облако = смесь джинн-газа с обычным → линия смешивания в (D/H, Y_p):
  D/H ↓, Y_p ↑, антикорреляция. НИ ОДИН астрофизический канал не даёт
  безметалльный газ, бедный D и богатый He (звёздная переработка сжигает D,
  но обязана давать металлы). Это самодостаточная улика, канал №1.

## Результаты пилотного анализа данных (июль 2026) — НЕ НАСТУПАТЬ НА ГРАБЛИ

1. **От одной петли нельзя поймать и GRB, и GW**: интервал прихода =
   T_cloud×(1+z) ~10⁷–10⁸ лет >> 50 лет архивов. Наблюдаемые пары:
   GW × постоянный абсорбер (позиция+z), популяционная статистика,
   GW→исчезновение только при τ(1+z) < десятилетий.
2. **GWTC — НЕ ТОТ КАТАЛОГ.** GWTC = слияния двойных (CBC). Сигнал коллапса
   петли — одиночный немоделированный ВСПЛЕСК (burst); правильный архив —
   burst-поиски (cWB all-sky). Их нуль на z<1 моделью ПРЕДСКАЗАН (петли
   локально подавлены падающей плотностью). Никогда не матчить облака с
   GWTC-событиями как с сигналами петель.
3. **z-разрыв**: громкие GWTC-события (SNR≥15, рингдаун тестируем — всего
   35 из 391) живут на z ≤ 0.7; низкометалличные абсорберы — на z ~ 2–4.5.
   Пересечение пустое. GW-канал петель открывается с Einstein Telescope /
   Cosmic Explorer. Не тратить время на GW×абсорбер с текущими детекторами.
4. Прогон 02_filter_clouds на 7 precision-D/H системах (Cooke et al. 2018,
   Table 3, значения проверены пересчётом log(D/H)→D/H — сходится):
   кандидатов на 3σ ноль, f_jinn < ~0.05 в этих облаках — первая точка
   ограничения. J1444+2919 (1.7σ ниже, f≈0.22, но σ огромная) — статус
   проверен: Cooke2018 §4.1 явно исключает его (и Riemer-Sørensen+2017,
   Zavarygin+2017) из Precision Sample из-за другой методики анализа
   (Balashev+2016, независимый пайплайн), не из-за качества данных.
   Держим в таблице отдельной строкой с флагом precision_sample=False.

## Пайплайн (этот репозиторий)

01_fetch_catalogs.py — качает GWTC CSV (справочно), Fermi GBM tdat, Swift BAT,
SDSS DR16 DLA (CDS/VizieR J/ApJS/258/18, .dat.gz + ReadMe — не Zenodo,
исходная посылка TODO была неверной; все 4 источника проверены, качаются).
02_filter_clouds.py — ГЛАВНЫЙ канал: отбор по «металлы отсутствуют + D/H ниже
стандарта», предсказание Y_p через линию смешивания для каждого кандидата;
плюс select_dla_targets() по полному DR16 DLA -> out/dla_targets.csv (см. TODO).
03_crossmatch.py — позиционный матч облака × короткие GRB (у GRB нет z; GRB
обязан приходить РАНЬШЕ света облака — обратный порядок отсеивать). Заготовка
in_credible_region под GW-skymaps эры ET. load_fermi_tdat() чинён 2026-07-07:
был баг с индексом при разборе имени поля (брался тип char11 вместо имени)
и с лишним хвостовым "|" в каждой строке данных — раньше падало с
AssertionError на любом реальном прогоне, сейчас 725 коротких GRB грузятся.
04_background_mc.py — фон скремблингом ТОЛЬКО по RA (Dec каждого GRB
фиксирован — сохраняет декл.-зависимую экспозицию GBM без построения
карты по poshist). Решение автора 2026-07-07: строгий poshist-фон
(десятки GB орбитальной истории) не делать — анизотропия экспозиции
это десятки процентов, тонет в градусных ошибках локализации и малой
выборке облаков. Строить, только если появится кандидат с p~0.01
и дело пойдёт к публикации совпадения.
05_ew_screen.py — Stage 2 сужения DLA-целей, ВЕРСИЯ 3 (matched-filter,
после двух отозванных версий, см. TODO): скан узкого окна интегрирования
EW (±100 км/с) с шагом 50 км/с по всему ±500 км/с для SiII 1526, CII 1334,
OI 1302, AlII 1670 -> out/dla_ew_screening.csv. SOLO-детекция: пик
одной линии >3.36σ (Sidak, n_eff=7). PAIR-детекция: 2 линии >2.0σ
одновременно, согласованные по скорости в ±70 км/с. Плюс информативность
предела (max_3sig_UL) на каждую запись. Фон, ~7900 целей, ~85 мин на
10 потоках (скан дороже box-car), инкрементальная запись.
06_merge_candidates.py — слияние Stage 1 (data/dla_metallicity_compilation.csv,
Rafelski+2012, MH<-2.0) + Stage 2 v3 с фильтром качества (3σ UL<0.2Å,
сопоставимо с калибровкой) -> out/final_candidates.csv, **N=39**,
отсортировано по потоку.

## v3-триггер: перенос пайплайна на DESI (начато 2026-07-09)

Задача от автора: расширить скрин с SDSS DR16 на DESI DR1, слить с текущими
39 в объединённый ранжированный список. Этапность обязательна (главный
риск — инфраструктура DESI, не метод); идти дальше только по отмашке.

**Этап 1, разведка — пройден, все три потенциальные стены проверены
фетчем, не по памяти:**
- Каталог: DESI DR1 DLA Toolkit, Brodzeller et al. 2025 (arXiv:2503.14740,
  Phys. Rev. D 112, 083510). URL проверен прямым скачиванием:
  `data.desi.lbl.gov/public/dr1/vac/dr1/dla-toolkit/v2.0/dlacat-dlatoolkit-dr1-main-dark-v2.0.fits`
  (13.8MB, размер совпал с листингом). **N=98766** DLA-кандидатов (SDSS
  Chabanier — 117458, тот же порядок). В заголовке статьи «DR2 Lyα BAO»,
  но Data Availability и путь файла подтверждают DR1 — не противоречие,
  DR2-BAO использует DR1-каталог как вход.
- Метод детектирования: **три** классификатора — Gaussian Process
  (Ho+2021) + CNN (Wang+2022) + новый spectral template classifier (введён
  самой этой статьёй), комбинированные с приоритетом GP при согласии.
  Не CNN-only, как у Chabanier — потенциально точнее относительно
  металл-линий, но это гипотеза для этапа 2, не факт (проверено на
  этапе 2, см. ниже).
- Спектры: сырое хранение — healpix coadd (nside=64 nested), ~219MB на
  файл, много целей внутри одного — bulk-фетчер, как у SDSS, был бы
  негодным. Но существует **SPARCL API** (NOIRLab, `sparclclient` на
  PyPI) с точечным доступом по TARGETID, минуя bulk-файлы; DESI-DR1 в
  списке датасетов. Проверено на реальных таргетах: `find()`+`retrieve()`
  батчем из 20 targetid -> 23 спектра (повторные наблюдения) за 29с =
  **~1.3с/спектр** — тот же порядок, что SDSS. Полный скрин на несколько
  тысяч целей остаётся в часах, не сутках. NB: `SparclClient()` по
  умолчанию `connect_timeout=1.1`s — слишком мало для этого окружения,
  нужно `connect_timeout=30` явно, иначе падает по таймауту на самом
  подключении.
- Пересечение с металл-редшифтами: кросс-матч 98766 DESI-DLA против 242
  систем Rafelski (тот же допуск 3″/0.02, что для SDSS) — **34 совпадения
  по абсорберам, 29 уникальных QSO** (SDSS давал 59). Меньше, но достаточно
  для реального измерения dv-систематики.

**Этап 2, dv-систематика на DESI — пройден
(`desi_01_fetch_dla.py`, `desi_02_dv_systematic.py` ->
`out/desi_zdla_vs_metal_velocity_offset.csv`):**
Вопрос был не «воспроизводится ли баг Chabanier», а «чинят ли методы
следующего поколения (GP+CNN+template) привязку Z_DLA к металл-линиям».
Ответ честный, с доверительными интервалами (для доли — Wilson, не
bootstrap: bootstrap на счётчике 0/34 всегда даёт [0,0], это артефакт
метода на нулевом счётчике, а не оценка неопределённости; для медианы —
percentile bootstrap, там счётчик ни при чём):

| | SDSS/Chabanier (n=59) | DESI/Brodzeller (n=34) |
|---|---|---|
| median \|dv\| | 79.5 [95% CI 50.1–113.8] км/с | 54.6 [95% CI 33.0–79.8] км/с |
| \|dv\|>300 км/с | 10.2% (6/59) [Wilson 4.7–20.5%] | 0.0% (0/34) [Wilson 0.0–10.2%] |
| max \|dv\| | 749 км/с | 280 км/с |

Точечные оценки у DESI лучше по обеим метрикам, но **доверительные
интервалы долей пересекаются** (DESI-верх 10.2% = SDSS-точка 10.2%) —
на n=34 нельзя статистически отличить «DESI действительно чинит
привязку» от «просто не наткнулись на хвост при такой выборке». Это
вторая точка на кривой эволюции систематики, но пока не доказательство
улучшения; нужен больший n (после этапа 3, за счёт роста пересечения
DESI×Rafelski по мере анализа, или отдельного метал-редшифт источника
для DESI) прежде чем утверждать «DESI решает проблему».

**Этап 3, matched-filter скрин DESI + слияние — пройден
(`desi_03_select_targets.py` -> `desi_04_ew_screen.py` ->
`desi_05_calibrate.py` -> `desi_06_merge.py`):**

- Порог целей обоснован цифрой (не назначен): NHI>=20.3 (конвенция SDSS)
  + **SNR_FOREST>=5** -> 4708 целей из 98766. Порог найден проверкой:
  SNR_FOREST>=5 сохраняет ~100% подтверждённых Rafelski-систем,
  SNR_FOREST>=7 уже теряет часть. **DELTACHI2 протестирован и ОТКЛОНЁН**
  как жёсткий фильтр — у подтверждённых систем минимум (0.023) ниже
  25-го перцентиля всей популяции, разделяющей силы не показал;
  отрицательный результат зафиксирован в коде, не тихо выброшен.
- Sanity-check на 100 случайных целях (правило 4, для нового фетчера и
  детектора) — пройден чисто: 100/100 фетч, 4/4 линий на каждой цели,
  медиана UL=0.161Å практически совпала с калибровкой SDSS (0.18Å),
  NaN/сюрпризов нет.
- Калибровка на 27 из 34 Rafelski-пересечений (7 отсеялись порогом
  SNR_FOREST/NHI — это цена строгого порога цели, ожидаемо): **0/27
  false-negative** [Wilson CI 0-12.5%], включая систему [M/H]=-2.54.
- Полный прогон (4708 целей, SPARCL batch): **66 сырых кандидатов** ->
  **38 после фильтра качества** (UL<0.2Å, конвенция SDSS). Сравнение
  долей с SDSS (453/7929=5.7% сырых -> 29/7929=0.37% финал):
  DESI сырая доля 66/4708=1.4% — в 4 раза НИЖЕ; но retention после
  UL-фильтра 38/66=57.6% — на порядок ВЫШЕ, чем у SDSS (29/453=6.4%).
  Финальная доля DESI (38/4708=0.81%) в ~2.2 раза ВЫШЕ, чем у SDSS
  (0.37%). Правдоподобное объяснение (не доказанное, гипотеза): порог
  цели у DESI строже с самого начала (SNR_FOREST>=5 против SDSS SNR>=3),
  поэтому в сырой кандидатский пул не попадает мусорный хвост
  малоинформативных нулевых детекций, который на SDSS составлял
  большинство из 453. НЕ обошлось без manual-inspection проверки, какую
  прошли SDSS-39 (визуальные PNG-панели, объективная перепроверка
  одиночным пикселем) — 38 DESI-кандидатов её пока не проходили,
  доверие к ним ниже, чем к уже проверенным SDSS-39, это явно
  зафиксировано как ограничение, не скрыто.
- Слияние: SDSS 39 + DESI 38 -> **77 в объединённом списке**, дублей по
  позиции+z вычищено **0** (кросс-обзорных совпадений не нашлось — см.
  ниже, это не 0 из-за отсутствия проверки, dup-check явный, прогнан).
  Кросс-обзорных ("золотых") целей: **0**. Не тревожный ноль: у DESI и
  SDSS шорт-листов (4708 и 7996) есть реальное пересечение по
  позиции+z — **1110 общих сайтлайнов** (~24% шорт-листа DESI), но
  при базовых частотах кандидатов 0.37% (SDSS) и 0.81% (DESI) ожидаемое
  число случайных двойных попаданий на 1110 пересечениях — <0.1
  (0.37%×0.81%×1110≈0.033), то есть 0 наблюдённых полностью согласуется
  со случайностью, а не сигнализирует о проблеме метода.
- Яркость — в собственных единицах каждого обзора (не сопоставимы
  напрямую): SDSS Flux медиана 5.73 (диапазон 0.97-21.7); DESI
  SNR_FOREST медиана 6.33 (диапазон 5.07-16.2, отражает уже применённый
  порог SNR_FOREST>=5).
- **Инженерные грабли SPARCL, для протокола** (три отдельных сбоя
  прогона, все исправлены до финального числа): (1) `SparclClient`
  хардкодит `MAX_CONNECT_TIMEOUT=3.1`s — параметр `connect_timeout`
  конструктора молча обрезается библиотекой, попытка увеличить его до 30
  не работает; (2) сама инициализация клиента (`__init__`) делает
  непойманный version-check GET с тем же потолком — уронила прогон на
  старте резюма без единой попытки; (3) resume/дедуп по одному TARGETID
  терял вторые абсорберы у 441 из 4267 целей с >1 DLA на одном
  сайтлайне (тот же класс, что precedent 92->66) — исправлено на ключ
  (TARGETID, zDLA). Обходной путь на все три: retry-обёртка на сетевые
  исключения (`RequestException`, `UnknownSparcl`, `ServerConnectionError`)
  вокруг и `get_client()`, и `fetch_batch()`, плюс per-batch try/except
  в `run()`, чтобы одна неудачная пачка не роняла весь прогон.

**Заготовка про кросс-обзорное ранжирование (см. выше) реализована**:

## Класс отказа: "правильное окно вокруг неправильного центра"
(named 2026-07-09, после кейса 83291431)

Три независимых инцидента проекта — разные механизмы, один и тот же
структурный сбой: **окно/ключ спроектированы верно, но привязаны к
величине, которую код молча считает точной, а она на самом деле сама
измерение со своей ошибкой (или недостаточна для уникальности)**.

1. **Box-car против CNN-сдвига** (отозванная версия 1 Stage-2, см. TODO
   выше). Окно ±300 км/с — разумный размер. Центр — zCNN, авторедшифт HI
   от нейросети Chabanier. zCNN считался "позицией линии", а на деле это
   independent измерение HI-профиля с собственной ошибкой относительно
   истинной скорости металлов (медиана 79 км/с, хвост до 749 км/с,
   out/zcnn_vs_metal_velocity_offset.csv). Окно верного размера, надетое
   на неверный центр, methodically мажет мимо реальной линии.
2. **Resume-дедуп по TARGETID без z_abs** (05_ew_screen.py /
   desi_04_ew_screen.py, найден при прогоне DESI). Здесь "окно" —
   множество уже обработанных ID, "центр" — сам TARGETID как единственный
   ключ. TARGETID корректно определяет сайтлайн, но НЕ абсорбер — 441 из
   4267 DESI-целей имеют >1 DLA на одном сайтлайне. Ключ спроектирован
   верно для 90% случаев и молча теряет данные там, где предположение
   "один TARGETID = одна запись" перестаёт быть точным.
3. **Межобзорный разъезд z_DLA** (кейс SDSS 83291431, этап 3.5). Окно
   ±500 км/с — уже с запасом, специально расширенное после первого
   инцидента. Центр — z_DLA САМОГО SDSS, который считался достаточно
   точным ориентиром на "эту систему". Но z_DLA DESI для той же системы
   разошёлся на ~415 км/с — окно снова верного размера, снова надето на
   центр, который сам был измерением, а не истиной, просто уровень
   абстракции сместился с "z_DLA vs z_metal" на "z_DLA(SDSS) vs
   z_DLA(DESI)".

**Формулировка защиты** (предложена автором 2026-07-09): центр окна —
и любой ключ, используемый для группировки/поиска — по умолчанию
считать измерением с собственной ошибкой или неполнотой, а не истиной,
пока не проверено обратное. Практическое следствие уже применённое в
проекте: коэффициент запаса на размер окна (±300 -> ±500 км/с) не решает
проблему сам по себе, если центр всё ещё может уехать на сопоставимое
расстояние (кейс 3 это доказал на уже расширенном окне) — нужна либо
проверка альтернативных центров (как PAIR-критерий проверяет
согласованность между линиями), либо независимая калибровка самого
центра (как этап 3.5 использует второй инструмент для перепроверки
z_DLA), а не просто увеличение окна.

## Правила проекта (прецеденты 2026-07-07, не косметика)

1. **Число не сообщается без проверки дублей.** Мёрж Stage1+Stage2 дал
   92 вместо 66 из-за join по ID без учёта z_abs (один QSO — несколько
   DLA-абсорберов на разных z, это норма каталога, не край случай).
   Поймано перепроверкой перед тем, как число ушло автору. Правило: любой
   count/merge по каталогу с потенциально неуникальным ключом — проверять
   на дубли явно, прежде чем печатать итог.
2. **Скрин не доверяется без калибровки на известных металличностях.**
   SDSSJ1419+0829 (precision-выборка, [O/H]=-1.92) — единственный
   precision-квазар в шорт-листе DLA — EW-скрин ложно объявил его чистым.
   n=1, но сигнал был верный: калибровка на 58 известных системах и
   ручная проверка топ-15 вскрыли, что это не редкий сбой, а системная
   поломка метода (58% ложных "чистых" на объективной перепроверке,
   см. TODO). Правило: ни один automated detection/non-detection скрин
   не публикуется как список кандидатов без прогона на выборке с ИЗВЕСТНЫМ
   ground truth (даже маленькой) — n=1 уже достаточно, чтобы насторожиться.
3. **Проверять то, что скрин РЕАЛЬНО измеряет, а не то, что он должен
   измерять по замыслу.** Box-car EW по широкому окну (±300 км/с) — это
   инструмент с конкретной, проверяемой чувствительностью, а не "измерение
   отсутствия линий" в общем смысле. Он слеп к узкой линии, утопленной в
   шуме остального окна, и к линии, сдвинутой за границу окна (zCNN ≠
   истинная скорость металлов). Оба эффекта подтверждены на конкретных ID
   с одиночными пикселями на 5-12σ, которые summary-статистика по всему
   окну не поймала. Правило: для любого нового automated-скрина сначала
   спросить "на каких физических эффектах он ломается формально корректным
   образом" (окно, редшифт, континуум), а не только "какая у него точность
   на бумаге".
4. **Исправление на одном примере — само по себе гипотеза, не факт.**
   Фикс "искать один самый глубокий пиксель вместо box-car" выглядел
   логично и прошёл калибровку (0% false-negative) — но при прогоне на
   реальных данных дал 868 кандидатов ВМЕСТО 57, то есть стал хуже, а не
   лучше: один пиксель — слишком шумный статистик, недобирал S/N линий,
   размазанных на 2-3 пикселя. Калибровка на known-metallicity выборке не
   поймала это, потому что калибровочные системы — сильные, хорошо
   детектируемые линии, они "прощают" плохую статистику. Правило: после
   любого фикса automated-метода — прогонять не только калибровку, но и
   sanity-check на случайной выборке РЕАЛЬНЫХ целевых данных (не только
   known-ground-truth), сравнивая долю кандидатов до/после — скачок в
   любую сторону (не только "меньше стало лучше") требует объяснения
   прежде, чем передавать число дальше.
5. **Число в публикацию сверяется с данными, а не с пересказом данных.**
   В §9 препринта v2 попало «калибровка на 58 системах» — но 58 было
   верно для ДРУГОГО, более раннего калибровочного прогона (box-car,
   с 1 fetch_failed на тот момент); финальная matched-filter калибровка,
   которую §9 реально описывает, дала n=59 (out/ew_screen_calibration.csv,
   49 metal-rich + 10 metal-poor). Опечатка родилась в CLAUDE.md (число
   верно для одного прогона, но CLAUDE.md — не машиночитаемый источник
   истины, это пересказ), и оттуда механически перекочевала в текст.
   Поймано постфактум пересчётом из самого CSV-файла, не заменой
   формулировок. Правило: любое число, уходящее в публикацию, проверяется
   последним прогоном по первичным данным (файл, а не абзац о файле) —
   особенно если в истории проекта было несколько одноимённых прогонов
   с разными результатами.
6. **«Запушено» — это состояние remote, прочитанное с remote, а не
   локальный git status.** После коммита `a118600` внешний инстанс со
   свежим клоном не увидел ни файла, ни правок в README — при том, что
   `git push` отчитался об успехе, а локальный `git branch -vv` показывал
   `[origin/main]` без расхождений. Перепроверка была сделана правильно
   (не поверить локальному статусу): `git fetch` + сравнение веток и
   GitHub REST API (`api.github.com/.../commits`, `.../git/trees`,
   `.../contents/...`) независимо подтвердили, что коммит и файлы реально
   на GitHub. Попутно поймана и конкретная техническая причина
   правдоподобного ложноотрицательного результата: `raw.githubusercontent.com`
   — это CDN с задержкой кэша (эмпирически — от нескольких десятков секунд
   до минут после пуша отдаёт старое содержимое), тогда как
   `api.github.com/repos/.../contents/...` читает git-блобы напрямую и
   видит изменения сразу же (проверено на следующем коммите: contents API
   показал новый текст, raw.githubusercontent.com в тот же момент — старый).
   Правило: утверждение "запушено" проверяется чтением
   удалённого репозитория напрямую (`git fetch` + `git log
   origin/main..main`, или REST/contents API — НЕ raw.githubusercontent.com
   и не кэширующийся веб-рендер, у обоих есть лаг), а не пересказом
   локального состояния — это применимо и к тому, КТО проверяет чужой
   пуш: проверка через кэширующийся веб-рендер не считается проверкой.
7. **Конкатенация разнотипных ID-колонок в pandas — молчаливая порча,
   не ошибка типа.** `desi_06_merge.py` склеивал DESI TARGETID (int64,
   до ~4×10¹⁶) и SDSS ID (float64 из каталога Chabanier) в одну колонку
   через `pd.concat`. Pandas апкастит всю колонку до float64 без
   предупреждения, а float64 точно хранит целые только до 2^53≈9×10¹⁵ —
   17-значные DESI TARGETID это давно превысили. Итог: 87.6% DESI
   TARGETID во всей популяции (86529/98766) и 34 из 38 текущих
   кандидатов молча портились при каждой записи в CSV. Сама находка
   этапа 3.5 (два кросс-обзорных контаминанта) не пострадала — матчинг
   шёл по ra/dec/z_abs, не по ID, — но колонка ID в закоммиченном файле
   была неверной до фикса. Правило: любая конкатенация/склейка ID-колонок
   из разных источников — привести оба к одному безопасному
   представлению (строка, если типы или диапазоны разные) ДО объединения,
   не после; молчаливый апкаст числового типа при `pd.concat`/`pd.merge`
   — тот же класс риска, что float64-округление координат (WebFetch,
   см. историю ra/dec), только источник ошибки внутри своего кода, а не
   у внешнего инструмента.

## v4-триггер (кандидат): архивное покрытие high-res спектроскопии
(Этап 1, 2026-07-10, `hires_01_archival_coverage.py`)

Задача: для 75 целей `out/merged_candidates_clean.csv` проверить конусным
поиском (5″), не сняты ли уже публичные high-/medium-res спектры — шанс
проверить недетекции без заявки на телескоп. Источники: KOA (Keck, клиент
PyKOA от NExScI) — HIRES (high) + ESI (medium); ESO (прямой TAP-запрос к
`archive.eso.org/tap_obs`, таблица `ivoa.ObsCore`) — UVES/ESPRESSO (high)
+ XSHOOTER (medium). Метаданные only, спектры не качались.

Rule 4 (сверка на реальных целях перед батчем) поймала два реальных
сюрприза до полного прогона:
1. Прокси в этом окружении интермиттентно 502-ит на обоих архивах
   (~30-40% запросов с первой попытки) — обычный retry с backoff,
   тот же паттерн, что SPARCL-прецедент в `desi_04_ew_screen.py`.
2. PyKOA может проглатывать сбой молча: при ошибке прокси иногда печатает
   сообщение об ошибке и возвращается БЕЗ исключения и БЕЗ записи
   выходного файла — обычный try/except-retry принял бы это за «нулевое
   покрытие» вместо «запрос не выполнился». Обёртка проверяет
   `os.path.exists`+размер файла явно, не только отсутствие исключения —
   ещё один случай «правильное окно вокруг неправильного центра»: окно —
   «исключения не было», ошибочно принятое за доказательство, что запрос
   прошёл.

SNR-колонка (добавлена по запросу автора после первого прогона) — только
где архив её реально отдаёт, не считается заново: KOA даёт `sig2nois`
(только для HIRES, у ESI такой колонки нет вообще — проверено живым
запросом, не пропущена), ESO даёт `snr` в `ivoa.ObsCore` (phase3-продукт).
Оба поля проверены на реальных данных проекта на предмет «не заглушка»:
`sig2nois` варьируется между экспозициями ОДНОЙ программы/ночи (цель
438888113, программа K293Hb: 6-9 на 9 экспозициях) — не фиксированное
номинальное число с обложки заявки; ObsCore `snr` варьируется между
продуктами одной цели (5.7-26.0 на 7 продуктах XSHOOTER). Решение —
прямой TAP, не `astroquery.eso`: раз `ivoa.ObsCore.snr` уже отдаёт
настоящие измеренные значения, обёртка `astroquery.eso` ничего не
добавляла. Напоминание автора про EXPTIME соблюдено с самого начала —
`total_exptime_s` пишется как есть, без фильтра, порог — задача этапа 2.

Флаг `covers_metal_lines` — необходимое, не достаточное условие: у
HIRES/UVES/ESPRESSO (эшелле) это общий диапазон blue-red экспозиции, а не
реальное покрытие по порядкам — линия может формально попасть в
[wave_min,wave_max] и всё равно провалиться в межпорядковый зазор;
настоящий ответ нужен по редуцированному решению по длинам волн, отложено
на этап 2. Флаг публичности: у ESO точный (`obs_release_date`), у KOA —
приближение (полусеместр из `semid` + `propint` месяцев, номинальные даты
конца полусеместра) за отсутствием прямой колонки в этой выдаче.

Resume: цель с нулём попаданий не оставляет строки в CSV — «нет в файле»
неотличимо от «ещё не обработана». Добавлен маркер-файл
`out/archival_coverage_done_ids.txt` (ID пишется туда после полной
обработки цели, независимо от результата) — пригодилось на практике:
первый полный прогон закончился чисто (75/75), но при добавлении
SNR-колонки повторный прогон был убит окружением на 15-й цели без
объяснения (`status: killed`, не исключение Python) — с маркер-файлом
третий прогон долетел до 75/75 без потери уже сделанной работы.

**Результат этапа 1:** 15 из 75 целей (11 SDSS + 4 DESI) имеют хотя бы
одно архивное попадание в радиусе 5″; 10 — high-res (HIRES/UVES/ESPRESSO);
все 15 — с хотя бы одной публичной экспозицией (проприетарных без
публичных пока не встретилось). По инструментам (число целей с
попаданием): ESI 11, HIRES 7, UVES 4, XSHOOTER 4. Ноль неудачных
запросов после ретраев. -> `out/archival_coverage.csv` (26 строк).
Ожидания были откалиброваны заранее (SDSS/DESI-цели отобраны по яркости
внутри своих обзоров, а не абсолютной — пересечение с историческими
Keck/VLT архивами могло быть от нуля до горстки) — вышла горстка.

**Дальше не идёт без отмашки автора** (этап 2 — скачивание и измерение
реальных спектров на найденных 15 целях — зависит от решения, стоит ли
это делать при текущем покрытии).

## v4-триггер, этап 2a: скачивание + инвентаризация (2026-07-11, `hires_02_inventory.py`)

Отмашка дана автором с явным сужением: приоритет 10 high-res
(HIRES/UVES/ESPRESSO), medium (ESI/X-Shooter) — второй эшелон и замена
там, где high-res не покрывает линию (531686697: AlII1670 из ESI).
Только публичные продукты, только экспозиции/продукты, покрывающие хотя
бы одну из 4 линий на z_abs цели; предпочтение phase3/reduced сырью.
Выход: `out/highres_inventory.csv` — реальный (не метаданный) диапазон
длин волн и SNR, измеренный по самому спектру у каждой линии, плюс флаг
пригодности к измерению. Резюмируемый прогон (тот же маркер-файл, что в
этапе 1) — пригодился: окружение убивало прогон дважды по ходу дела.

**Находка, закрывшая часть плана до скачивания:** у ESI в KOA **нет
level-1 (редуцированного) продукта вообще** — проверено живым запросом
(`Koa.download(..., lev1file=1)` вернул `"Instrument [ESI] does not have
level1 data."`, прямой ответ системы). Есть только сырые 2D-эшеллограммы;
превратить их в 1D-спектр с волновой калибровкой — значит написать
пайплайн редукции (трассировка порядков + калибровка длин волн +
экстракция), что вне объёма «скачивание+инвентаризация» и не делается
без калибровочной выборки (правило 2). Эти цели/линии идут в инвентарь
строкой `suitability=raw_only_no_reduced_pipeline`, без скачивания
бесполезных сырых кадров.

**Инженерная оптимизация (13×) после живого профилирования, а не по
предположению:** `Koa.download(lev1file=1)` тянет ВСЕ под-продукты lev1
(flux/hdr/arcids/trace/profile) по одному HTTP-запросу на файл без
фильтра — 233 файла на одну реальную экспозицию, из которых нужны только
17 (`flux`, с колонками wave/Flux/Error/Sig_to_Noise на пиксель). Прямые
запросы к тем же эндпойнтам, что использует сам pykoa внутри
(`nph-getL1list`/`nph-dnloadL1data`, проверено по исходнику pykoa и живым
запросом, файлы байт-в-байт идентичны) с фильтром по `flux`-подкаталогу —
сократили время на цель с ~14 минут до ~3.5 минут на том же тестовом
таргете. Без этой оптимизации полный батч занял бы часы, а не десятки
минут.

**Три находки на живых данных, ни одна не поймана бы по метаданным:**
1. **`covered_but_unmeasurable`** — AlII1670 у 387205842 (HIRES, ордер
   ccd3/07) формально попадает в номинальный диапазон ордера, но реальных
   пикселей у линии — 0 (край ордера, почти все пиксели помечены плохими,
   err=-1). Ровно тот класс отказа «правильное окно вокруг неправильного
   центра», который назвали в этапе 3.5 DESI — теперь пойман on the spot
   по самим данным, не постфактум. `suitability` различает «линия
   формально в диапазоне» от «реально измерима» (npix≥3 у хотя бы одной
   линии).
2. **`unsupported_lev1_format`** — часть HIRES-экспозиций отдаёт lev1 в
   старом макете (`makee/ccdN/fits`: отдельные Flux/Var/Sky/Arc/profile
   мульти-экстеншн FITS-картинки по порядку, без таблицы wave+flux+err)
   вместо `binaryfits/ccdN/flux`. Не привязано к дате: у 407352752 обе
   экспозиции (ночь 2010-04-04) — makee, а у 387205842 экспозиции ТОЙ ЖЕ
   ночи — binaryfits и прекрасно читаются. Читать второй формат не стали
   (нужна отдельная реконструкция волнового решения, неоткалибровано,
   тот же риск, что и ESI-редукция) — честно помечено, не молчаливый 0.
3. **`no_wavelength_solution`** — у 531686697 (экспозиция
   HI.20080329.39341, все 17 ордеров) колонка `wave` целиком забита
   сентинелом -1.0, при этом `Flux` содержит настоящие варьирующиеся
   значения — MAKEE проэкстрагировал поток, но калибровка длин волн для
   этой экспозиции не удалась. Подтверждено свежим скачиванием файла
   байт-в-байт заново (не проблема оборванной закачки: gzip целый,

## v4-триггер, этап 2b: измерение (2026-07-11, `hires_03_measure_purity.py`)

Отмашка дана автором. Метод — переиспользование `find_peak()`/`classify()`
из `05_ew_screen.py` БЕЗ ИЗМЕНЕНИЙ (тот же matched-filter скан, те же
пороги Z_SOLO=3.36/Z_PAIR=2.0/TOL_DV_PAIR=70км/с), чтобы числа были
сопоставимы с SDSS/DESI-медианами напрямую — не новый метод, тот же
протокол v3 на реальном поштучном error-спектре вместо ivar/and_mask
конвейера SDSS/DESI (ivar реконструирован как 1/err², and_mask — все
нули, err<=0 уже исключается через ivar>0 внутри find_peak). EW детекции
= sigma*ew_err_rest (инвариант при переходе в rest-frame, вывод без
модификации find_peak); EW верхнего предела = 3*ew_err_rest, та же
конвенция, что в `05_ew_screen.py`/`final_candidates.csv`. Один продукт
на линию — с наибольшим SNR из этапа 2a среди usable-продуктов,
покрывающих линию (не co-add нескольких экспозиций — проще и
консервативнее, но не максимально чувствительно, явно фиксируется как
выбор, не молчаливое допущение). AlII1670 без покрывающего продукта —
`status=unavailable` с причиной, не нулевой предел (по прямому указанию
автора); отдельно различены две причины unavailable: «ни один продукт не
покрывает линию» и «линия номинально в диапазоне, но <3 валидных
пикселей везде» (тот же `covered_but_unmeasurable` из этапа 2a, только на
уровне одной линии внутри иначе usable продукта — поймано багом
`idxmax()` на все-NaN срезе SNR, исправлено до финального прогона).

Класс сверки зависит от того, что цель реально заявляла (прецедент
`desi_08_exclude_contradicted.py`): **Stage2** targets (SDSS/DESI
matched-filter «0/4 линий») — detected → `contamination`, не detected
(n_ok>=3) → `confirmed_clean`, n_ok<3 → `ambiguous`. **Stage1** targets
(литературная [M/H]<-2, Rafelski) — детекция ОЖИДАЕМА, не противоречие:
detected → `consistent_with_literature`, not detected →
`unexpected_nondetection`.

**Итог: `out/highres_purity.csv`, 40 строк (10 целей × 4 линии).**
По статусу: `detected` 29, `upper_limit` 9, `unavailable` 2. По классу
сверки (на цель): `consistent_with_literature` 6 (все Stage1 — ожидаемо),
**`contamination` 4 из 4 Stage2-целей — 0 confirmed_clean**.

**Главная находка: 100% (4/4) usable Stage2-целей, дошедших до
измерения, оказались контаминированы на реальном high-res спектре.**
n=4 маленькое, обобщать на весь метод рано, но результат качественно
согласуется с тем, что проект уже знает про свой Stage-2 скрин
(box-car-отзыв, order-gap кейс этапа 2a) — низкое разрешение/SNR SDSS/DESI
систематически недооценивает слабые линии, matched-filter на BOSS-разрешении
по-прежнему пропускает то, что видно на Keck/VLT:
- `39627762889133674` (DESI/Stage2) — все 4 линии детектированы
  ОДНОВРЕМЕННО на dv=+500 км/с (SiII 9.1σ, CII 55.1σ, OI 12.7σ,
  AlII 17.5σ) — не могло быть случайным совпадением 4 независимых
  шумовых пиков на одной скорости. **Важная оговорка**: dv=500 км/с — это
  КРАЙ окна сканирования (SEARCH_KMS=500, унаследовано из SDSS/DESI-
  протокола без изменений) — реальное смещение может быть БОЛЬШЕ,
  скан упёрся в границу, а не нашёл истинный пик. То, что все 4 линии
  сошлись ровно на границе — сильный признак систематической ошибки
  z_abs у этой цели (тот же класс, что в DESI Stage 3.5, кейс 83291431),
  а не многокомпонентной структуры (та давала бы разные dv по линиям).
- `39628029219048108` (DESI/Stage2) — OI 21.5σ на dv=-200 (сильная,
  самостоятельная), плюс согласованная пара CII+AlII на dv=+350
  (2.6σ/45.7σ) — два разных компонента по скорости, оба реальны.
- `39627875648802065` (DESI/Stage2) — AlII 10.2σ на dv=+150, единственная
  видимая линия, но статистически надёжная сама по себе.
- `438888113` (SDSS/Stage2) — SiII 9.6σ (dv=0) и AlII 5.0σ (dv=-50),
  независимые SOLO-детекции на близких скоростях.

**Побочное подтверждение известной систематики**: среди 29 детекций
10 имеют |dv|>300 км/с от каталожного z_abs — включая Stage1-цели
(146393570 SiII dv=-450, 387205842 SiII dv=-450, 77863991 OI dv=-400,
219792523 CII dv=+400), где детекция сама по себе не сюрприз, но
величина смещения — количественное подтверждение systematики
zCNN/z_DLA-vs-metal-velocity, которую проект уже документировал
(out/zcnn_vs_metal_velocity_offset.csv, медиана 79 км/с, хвост до
749 км/с) — теперь на независимом, много более точном методе (реальные
Keck/VLT спектры, не совмещение каталогов).

**Два unavailable**: 387205842/AlII1670 (нет покрывающего продукта —

## Диагностика CII1334 (2026-07-11, `hires_05_cii_diagnostics.py`)

Отмашка автора, два конкретных вопроса. Разрешено оба:

1. **Локальная значимость на dv=+650** (не максимум скана, а значение
   именно в той точке, где сошлись остальные 3 линии) — переиспользован
   внутренний расчёт `find_peak()` для одной пробной точки: **125.67σ**,
   почти вровень с «максимумом» скана (133.95σ на dv=-1500). Это значит
   CII1334 РЕАЛЬНО согласуется с системой на +650 км/с — просто максимум
   скана оказался перебит другим, ещё чуть более значимым пиком в другом
   месте окна.

2. **Что на dv=-1500** (наблюдаемая λ=4886.24 Å): сверка с каталогом
   DESI DLA Toolkit (`out/desi_targets.csv`) по этому же сайтлайну

   Не телурика (4886 Å — не телурическая полоса), не склейка эшелле-
   порядков (это цельный UVB-продукт с QUAL=0 по всему окну, реальный
   провал потока до почти нуля на ~7 Å/430 км/с — слишком широкий для
   единичного плохого пикселя). Кросс-обзорная сверка v3: SDSS-покрытия
